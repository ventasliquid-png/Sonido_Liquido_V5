RIESGOS Y CONFLICTOS DETECTADOS
═══════════════════════════════════════════════════════════════════════════════

RIESGO 1: RACE CONDITION (409 CONFLICT) - CRITICO
- Problema: Dos uploads simultaneos del mismo PDF pueden generar duplicados
- Ubicacion: service.py lineas 30-51 (Anti-Zombi checks)
- Limitacion: Checks no son atomicos (lectura -> creacion puede fallar)
- Solucion: Transaccion con LOCK o UUID generado por cliente

RIESGO 2: FLAGS_ESTADO V14 vs V15 - ALTO
- Problema: Semantica de Bit20 cambio entre V14 y V15
- Ubicacion: flags_estado es BIGINT en facturas
- Impacto: Ingesta no valida el esquema version
- Mitiga: Conserje READ-ONLY no debe modificar flags

RIESGO 3: DOMICILIOS LEGACY vs NUEVO - ALTO
- Problema: Coexisten Domicilio (legacy) y VinculoGeografico (nuevo)
- Ubicacion: remitos/service.py linea 13 (ambas importadas)
- Impacto: Conserje debe soportar ambos modelos

RIESGO 4: OCR FALLA - MEDIO
- Problema: parse_invoice_data() puede fallar o extraer incompleto
- Ubicacion: pdf_parser.py linea 239 (validacion basica)
- Mitiga: Usuario edita manualmente en frontend

RIESGO 5: PDF MALFORMADO - MEDIO
- Problema: fitz lanza Exception generica sin detalles
- Ubicacion: pdf_parser.py linea 43
- Impacto: Usuario ve error opaco

RIESGO 6: AUDITORIA INCOMPLETA - MEDIO
- Problema: No se registra quien, cuando, que heuristicas
- Ubicacion: Sin tabla ni campo audit en facturas
- Mitiga: notas_auditoria (nuevo en 029) pero no usado

RIESGO 7: FALTA ATOMICIDAD INGESTA-PROCESS - MEDIO
- Problema: Si falla INSERT Remito despues de crear Pedido, queda zombi
- Ubicacion: service.py, create_from_ingestion (sin savepoint)
- Mitiga: Usar transacciones explicitas en ingesta v2

- Race condition sin lock
- Logica Conserje inline, no reutilizable


PROPUESTO (CON CONSERJE MODULAR):
PDF -> POST /ingesta/raw (store PDF como BLOB)
   -> INSERT facturas_raw
   -> Retorna: {raw_id, status:'RAW'}
-> GET /ingesta/raw/{raw_id}/preview (parse in-memory, READ-ONLY)
   -> Extraer texto con fitz
   -> Parse con heuristica Conserje
   -> Resolve cliente (CUIT + nombre fallback)
   -> Score domicilios
   -> Valida schema
   -> LOG: decision + confidence
   -> Retorna: {parsed_data, domicilios, audit}
   -> NO INSERT TODAVIA
-> USUARIO EDITA EN FRONTEND
-> POST /ingesta/raw/{raw_id}/process (validate + store parsed)
   -> Revisa cambios del usuario
   -> Re-valida schema
   -> INSERT facturas_procesadas (estado='PROCESADA')
   -> Retorna: {proc_id, status:'PROCESADA'}
-> GET /ingesta/procesadas/{proc_id} (audit trail READ-ONLY)
   -> Ver decisiones Conserje, confidence, timestamps
-> POST /remitos/desde_procesada/{proc_id} (create remito)
   -> Lee FacturaProcesada.parsed_data
   -> Convierte a IngestionPayload
   -> Llama RemitosService.create_from_ingestion()
   -> INSERT Remito, Pedido, Items
   -> UPDATE facturas_procesadas.estado='REMITO_CREADO'
   -> Retorna: {remito_id, status:'REMITO_CREADO'}

VENTAJAS:
- Historial completo: RAW -> PROCESADA -> REMITO
- Auditoria: cada decision logged con confidence
- Separacion de concerns: ingesta != remitos
- Conserje modular, reutilizable
- Preview sin comprometer datos
- Rollback seguro: DELETE raw (cascada limpia)
- Race condition resuelta con PK (raw_id)
- Transparencia: usuario ve por que se sugierio domicilio X
